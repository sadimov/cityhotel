Résumé exécutif

Synthèse de l'audit

La plateforme LiasseFiscale est une application de gestion des déclarations fiscales développée pour la Direction Générale des Impôts de Mauritanie. L'audit a été conduit en avril 2026 par l'équipe ADIAS dans le cadre de son intégration comme module du Système de Télédéclaration Tributaire (STT).

L'application présente une architecture globalement bien structurée, avec une séparation claire des responsabilités, un modèle de données riche couvrant les 3 catégories de liasse fiscale (G, B, F) et une couverture fonctionnelle complète du processus de déclaration. Cependant, plusieurs points de vigilance critiques doivent être traités avant l'intégration en production.

Constatations clés

  • Authentification HTTP Basic : mécanisme inadapté à la production, incompatible avec STT (prd.alis_user / JWT)
  • Credentials localStorage : exposition des identifiants côté client, vulnérable aux attaques XSS
  • Tests automatisés absents : couverture quasi nulle (1 seul fichier placeholder)
  • Flyway désactivé : risque de divergence de schéma entre environnements
  • CORS trop permissif : plages réseau privées entières autorisées (192.168.x.x, 10.x.x.x)
  • Logging commenté : traçabilité opérationnelle insuffisante, audit trail absent
  • Services incomplets : SMS (TWILIO/ORANGE stubs), OTP non intégré, PDF déclaration absent
  • Secrets en dur : credentials SMTP et JWT par défaut dans application.yml
  • Modèle de données complet : 54 entités JPA couvrant les 3 catégories fiscales
  • Spring Boot 4 / Java 17 : stack moderne, compatible avec l'écosystème STT
  • Séparation des responsabilités : architecture en couches bien respectée

Méthodologie

L'audit a été conduit selon une approche en 4 axes :

  1. Analyse statique du code — lecture complète du code source backend et frontend
  2. Revue architecturale — évaluation de la conception, des patterns utilisés et de leur cohérence
  3. Audit de sécurité — vérification des mécanismes d'authentification, de la gestion des secrets, des CORS, et des vulnérabilités OWASP
  4. Analyse d'intégrabilité — évaluation de la compatibilité avec la base de données STT et son modèle d'authentification

Architecture générale

Vue d'ensemble

LiasseFiscale suit une architecture 3-tiers classique avec séparation nette entre le frontend Angular (SPA), le backend Spring Boot (API REST) et la base de données PostgreSQL.

Schéma de déploiement actuel :
Navigateur (Angular 18 :8083) → Proxy → Spring Boot API (:9001) → PostgreSQL (schéma: liasse)

Le frontend utilise un proxy.conf.json qui reroute les appels /api/* vers le backend. En production, un reverse proxy (Nginx/Apache) est supposé gérer cette redirection.

⚠️ Absence de versionning API
Aucun préfixe de version (/v1/) n'est présent sur les endpoints. Lors de l'intégration dans STT, la coexistence avec d'autres APIs rendra la gestion des versions impossible sans refactoring.

Services métier

⚠️ ExcelImportService — Couverture partielle
L'import Excel supporte BilanActif, BilanPassif, ResultatComptable, BilanBanques, ResultatComptableBanques, InteretsBanques, DADS, ChargesReportées, Identification, ProvisionsBanques, CreancesRadiees. Les formulaires Immobilisations, Balance, Financement, AffectationRésultat, PrixTransferts ne sont pas parsés depuis Excel — uniquement accessibles via saisie manuelle.
→ À compléter pour une couverture Excel complète des 3 catégories
ℹ️ PdfService — Récépissés uniquement
Le service PDF ne génère que les récépissés d'inscription. Il n'existe pas d'export PDF des déclarations complètes (endpoint GET /declarations/{id}/pdf manquant). Cette fonctionnalité est attendue par les utilisateurs.
→ Implémenter l'export PDF complet de déclaration dans la prochaine itération
⚠️ SmsService — Providers TWILIO/ORANGE non implémentés
Le SmsService dispose d'une architecture à 3 providers (LOCAL, TWILIO, ORANGE) mais seul le provider LOCAL est implémenté. Les providers TWILIO et ORANGE sont des stubs vides (// TODO). De plus, les logs sont commentés — en cas d'échec d'envoi SMS, aucune trace n'est conservée.
→ Implémenter le provider mauritanien approprié et restaurer le logging

Authentification côté client

🔴 CRITIQUE — Credentials Base64 stockés dans localStorage
L'intercepteur HTTP (jwt.interceptor.ts) encode les identifiants en Base64 et les stocke dans localStorage sous la clé dgi_credentials. Le localStorage est accessible par tout JavaScript de la page, rendant ces credentials vulnérables aux attaques XSS. Un seul script malveillant peut extraire l'ensemble des identifiants de l'utilisateur.

Impact : Compromission complète d'un compte utilisateur (y compris SUPER_ADMIN) via XSS.
→ Migration vers JWT avec HttpOnly cookies (non accessible en JavaScript) lors de l'intégration STT est OBLIGATOIRE

Gestion d'état et i18n

⚠️ i18n câblé mais non branché
Le service i18n.service.ts et la bibliothèque @ngx-translate/core sont présents. Les fichiers de traduction sont dans src/i18n/. Cependant, aucun composant n'utilise le pipe translate — l'application est entièrement en français hardcodé. Le support arabe (AR) — requis pour la DGI mauritanienne — n'est pas actif.
→ Activer l'i18n avec support RTL arabe lors de l'intégration STT
ℹ️ OtpComponent — Présent mais non routé
Le composant OTP (otp.component.ts) est présent mais non inclus dans le routeur. L'infrastructure OTP (Google Authenticator) est préparée côté backend (OtpService.java) mais non intégrée au flux de login.
→ Intégrer l'OTP comme second facteur lors de l'intégration STT pour les comptes ADMIN+

Migrations Flyway

🔴 CRITIQUE — Flyway désactivé en production
La configuration spring.flyway.enabled: false désactive complètement les migrations automatiques. Le schéma doit être appliqué manuellement via les 16 scripts V1–V16. En environnement multi-instance ou lors d'un déploiement STT, ce risque de divergence de schéma peut entraîner des erreurs silencieuses au runtime (ddl-auto: validate lèvera des exceptions si le schéma ne correspond pas).
Impact : Downtime en cas de migration manuelle oubliée, perte de données potentielle.
→ Réactiver Flyway avec enabled: true et locations: classpath:db/migration avant l'intégration STT

⚠️ Duplication des données contribuable
La table liasse.contribuable duplique des données déjà présentes dans taxpayer.taxpayer (STT) : raison sociale, adresse, forme juridique, kserv. Cette duplication est une source de divergence potentielle. Le NIF est la seule clé commune exploitée actuellement.
→ Lors de l'intégration, la table liasse.contribuable devrait être remplacée par une vue ou une foreign key vers taxpayer.taxpayer

CORS et headers HTTP

⚠️ CORS — Plages réseau privées trop larges
La configuration CORS autorise http://192.168.*.*:* et http://10.*.*.*:* — ce qui couvre l'ensemble des réseaux privés RFC-1918. En environnement de bureau partagé ou dans un bureau DGI, n'importe quelle machine du réseau local pourrait émettre des requêtes Cross-Origin authentifiées vers l'API.

Extrait de configuration :
→ Restreindre les origines autorisées aux seules URLs connues. En production : uniquement https://liassefiscale.impots.gov.mr et les domaines STT.
⚠️ Headers de sécurité HTTP absents
Les headers de sécurité recommandés ne sont pas configurés : X-Content-Type-Options, X-Frame-Options, Strict-Transport-Security (HSTS), Content-Security-Policy. Ces headers protègent contre le clickjacking, MIME sniffing, et les injections.
→ Ajouter ces headers via Spring Security .headers() ou au niveau Nginx/reverse proxy

Gestion des secrets

🔴 CRITIQUE — Credentials SMTP en dur dans application.yml
Le fichier application.yml contient des credentials SMTP réels par défaut :
username: sasidimed@gmail.com
password: gryxpjdjevaiuhrn

Ces valeurs sont dans le dépôt de code source. Si le dépôt n'est pas privé ou si un développeur quitte l'équipe, ces credentials sont compromis.
→ IMMÉDIAT : Remplacer par des variables d'environnement (${MAIL_USER}, ${MAIL_PASS}). Révoquer et régénérer les credentials actuels. Ne jamais commettre de secrets dans le code.
⚠️ Clé JWT par défaut dans le code source
La clé secrète JWT est définie en valeur par défaut dans application.yml : VGhpcyFpcyFhIXZlcnkhbG9uZyFzZWNyZXQha2V5IWZvciFsaWFzc2VGaXNjYWxlITIwMjQ=. Bien que JWT ne soit pas activé actuellement, si l'activation se fait sans changer cette clé, tous les tokens seront signés avec un secret public.
→ Configurer JWT_SECRET uniquement via variable d'environnement sans valeur par défaut dans le code

Qualité du code et maintenabilité

Points forts

  • Architecture en couches respectée (controller → service → repository → entity)
  • DTOs records Java 17 — immutables, concis et sûrs
  • ApiResponse<T> unifié avec factory methods ok() / error()
  • Annotations Spring bien utilisées (@Service, @Repository, @Transactional)
  • GlobalExceptionHandler centralisé pour la gestion des erreurs
  • Angular standalone components — architecture moderne
  • Services Angular injectables avec signals pour l'état utilisateur

Points d'amélioration

  • Couverture de tests quasi nulle — 1 seul fichier placeholder LiassefiscaleBackendApplicationTests.java
  • Logs commentés dans EmailService et SmsService — traçabilité opérationnelle absente
  • Usage incohérent de Lombok — certaines entités utilisent @Data, d'autres ont des getters/setters explicites
  • DeclarationWizardComponent monolithique (~3 361 lignes) — difficulté de maintenance
  • Certaines entités avec @RequiredArgsConstructor sans @NoArgsConstructor pour JPA (problèmes potentiels avec les proxies Hibernate)
  • Pas de documentation Swagger complète sur tous les endpoints

Performance et scalabilité

Backend

  • HikariCP pool : max 10 connexions, min 2 — correct pour une charge modérée
  • Opérations asynchrones (@Async) pour emails et SMS — non bloquant pour l'utilisateur
  • Import Excel en @Async — traitement non bloquant
  • Cache Spring simple (type: simple) — en mémoire, non distribué. Insuffisant si plusieurs instances
  • Pas de pagination sur tous les endpoints listes (FormController retourne des listes complètes)
  • Pas de compression GZIP configurée sur les réponses API
  • Pas de cache HTTP (Cache-Control headers) sur les ressources statiques frontend

Base de données

  • Indexes sur les colonnes de jointure (kpm, nif) à vérifier via les migrations
  • Relations OneToOne CascadeType.ALL sur tous les sous-formulaires — suppression en cascade potentiellement dangereuse
  • Chargement LAZY non configuré explicitement sur les relations — risque N+1 queries
  • Pas de partitionnement de table pour les grandes déclarations

Préparation à l'intégration STT

Compatibilités confirmées

  • PostgreSQL 15.2 — même instance, ajout du schéma liasse dans la base postgres STT
  • Java 17 — compatible avec l'environnement JVM STT
  • Infrastructure JWT préparée (jjwt 0.12.6 dans pom.xml) — prête à activer
  • NIF comme clé métier commune — liasse.contribuable.cmatrfisc = taxpayer.nif.nif
  • i18n (@ngx-translate) installé — prêt à activer pour FR/AR

Incompatibilités à résoudre

  • Authentification : HTTP Basic → STT utilise JWT / prd.alis_user
  • Gestion des utilisateurs : table db_user → fusion avec prd.alis_user
  • Design system : Tailwind + Bootstrap → design system STT unifié
  • Structure des divisions : liasse.structure → referential.division (STT)
  • Données contribuable : liasse.contribuable → vue sur taxpayer.taxpayer (STT)
  • CORS : restrictions à durcir pour l'environnement STT

Recommandations prioritaires

P0 — Blocants (avant tout déploiement)

1. Remplacer HTTP Basic Auth par JWT avec HttpOnly cookies
L'infrastructure JWT est déjà présente dans le pom.xml (jjwt 0.12.6). Il faut l'activer et configurer Spring Security pour émettre des JWT signés stockés en HttpOnly cookie.
Effort estimé : 3–5 jours développeur · Intégration avec prd.alis_user STT recommandée
2. Retirer les secrets du code source
Remplacer immédiatement les credentials SMTP hardcodés dans application.yml. Utiliser exclusivement des variables d'environnement ou un vault (HashiCorp Vault, Spring Cloud Config).
Effort estimé : 1 heure · IMMÉDIAT — Révoquer et régénérer les credentials actuels

P1 — Haute priorité (avant intégration STT)

3. Réactiver Flyway et versionner les migrations
Réactiver spring.flyway.enabled: true. S'assurer que tous les scripts V1–V16 sont dans l'ordre correct et testés sur une instance fraîche. Configurer le chemin liasse pour Flyway dans la base STT.
Effort estimé : 1–2 jours · Critique pour les déploiements automatisés CI/CD
4. Implémenter le rate limiting sur les endpoints d'authentification
Ajouter Spring Security Bucket4j ou un filtre de rate limiting sur /api/auth/login, /api/auth/password/reset-request. Limiter à 5 tentatives / minute / IP.
Effort estimé : 1 jour · Alternative : configurer au niveau Nginx/reverse proxy
5. Durcir la configuration CORS
Supprimer les plages 192.168.x.x et 10.x.x.x des origines autorisées. En production : uniquement les domaines STT et DGI connus.
Effort estimé : 30 minutes

P2 — Importante (lors de l'intégration)

6. Ajouter les headers de sécurité HTTP
Configurer dans SecurityConfig.java ou Nginx : X-Content-Type-Options: nosniff, X-Frame-Options: DENY, HSTS, CSP.
Effort estimé : 2 heures
7. Restaurer les logs et implémenter un audit trail
Décommenter et enrichir les logs dans EmailService, SmsService. Créer une table audit_log pour tracer les actions critiques : login, validation déclaration, création d'utilisateur, reset mot de passe.
Effort estimé : 2–3 jours
8. Découper le DeclarationWizardComponent
Décomposer le composant monolithique (~3 361 lignes) en sous-composants Angular par étape du wizard. Améliore la maintenabilité et facilite le test unitaire.
Effort estimé : 5–8 jours

Conclusion

Verdict d'audit

La plateforme LiasseFiscale est fonctionnellement complète et bien architecturée pour une application de première génération. Le modèle de données est riche et couvre l'ensemble des catégories fiscales. Cependant, l'application présente des lacunes de sécurité importantes qui la rendent non prête pour un déploiement en production immédiat dans l'environnement STT sans corrections.

Les deux points critiques — le stockage des credentials en localStorage et les secrets SMTP en dur dans le code — doivent être corrigés avant toute intégration. La migration vers le système d'authentification STT (prd.alis_user + JWT) est la transformation la plus structurante et devra être planifiée soigneusement.

Recommandation de l'équipe ADIAS : Procéder à l'intégration selon le plan en 3 phases décrit dans le document complémentaire « Plan d'Intégration STT », avec une phase de sécurisation en priorité absolue.
